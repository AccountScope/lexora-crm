EXECUTIVE SUMMARY
================================================================================

Lexora is a production-ready legal practice management platform built for UK 
solicitors and barristers. It includes case management, SRA-compliant trust 
accounting, document vault, time tracking, billing, and client portal.

The platform has been built through 4 major development phases over 4 weeks:
- Phase 1: Core legal CRM functionality
- Phase 2: Professional UX polish (tooltips, formatting)
- Phase 3: Interactive onboarding (tours, help system)
- Phase 4: Advanced features (videos, AI chat, gamification)

TECHNICAL ARCHITECTURE
================================================================================

FRONTEND:
- Framework: Next.js 14 (App Router)
- Language: TypeScript (strict mode)
- Styling: Tailwind CSS + shadcn/ui
- State Management: TanStack Query (React Query)
- Forms: React Hook Form + Zod validation
- Charts: Chart.js
- Icons: Lucide React

BACKEND:
- Database: Supabase (PostgreSQL)
- Authentication: Supabase Auth (email/password, OAuth)
- Storage: Supabase Storage (documents)
- API: Next.js API Routes (serverless)
- PDF Service: GCP Cloud Run (separate service)

INFRASTRUCTURE:
- Hosting: Vercel (production + preview)
- Database: Supabase (managed PostgreSQL)
- CDN: Vercel Edge Network
- Version Control: Git (committed locally)

KEY LIBRARIES:
- cmdk (⌘K command palette)
- @radix-ui/* (UI primitives)
- date-fns (date formatting)
- react-hook-form (forms)
- zod (validation)
- chartjs-node-canvas (PDF charts)

KNOWN ISSUES & WARNINGS
================================================================================

1. Dynamic Server Usage Warnings (SAFE TO IGNORE)
   Multiple API routes show warnings about dynamic server usage.
   These are expected for API routes that use request.headers.
   They do not block production deployment.
   Example:
   "Route /api/activity couldn't be rendered statically..."

2. npm Audit Vulnerabilities (13 total: 5 moderate, 8 high)
   These are in dev dependencies and do not affect production.
   Can be addressed with: npm audit fix --force
   (May cause breaking changes, test thoroughly after)

3. TypeScript Build Info (3.2MB file)
   Large .tsbuildinfo file is normal for TypeScript incremental builds.
   Excluded from deployment automatically.

4. .next Build Folder (1.5GB)
   Large build folder is normal for Next.js with many pages/routes.
   Vercel optimizes this during deployment.
